# GSharp web app

Generated by `dotnet new gsharp-web`.

This minimal [ASP.NET Core](https://learn.microsoft.com/aspnet/core/) web application uses the modern `WebApplication` host (Kestrel) and a terminal request handler that responds with `Hello from GSharp on ASP.NET Core!` for every request. The handler is declared as a `RequestDelegate` value so the GSharp function literal converts cleanly to the delegate the ASP.NET pipeline expects, and it is registered as terminal middleware with `app.Run(handler)`. The response body is written asynchronously through `HttpResponse.WriteAsync`.

The project references the shared `Microsoft.AspNetCore.App` framework via a `<FrameworkReference>` in `GsharpWebApp.gsproj`, so no extra NuGet packages are needed for the web stack.

## Build & run

```sh
dotnet build
dotnet run
```

Then open <http://localhost:5000/> in a browser or `curl` it. Press <kbd>Ctrl</kbd>+<kbd>C</kbd> to stop.

On macOS, port 5000 is used by the AirPlay Receiver service; change the URL in `Program.gs` (for example to `http://localhost:5050`) if the port is already in use.

## What the compiler supports today

This template demonstrates that GSharp can drive the ASP.NET Core pipeline using language features that now work end to end:

- Constructing and configuring imported types (`WebApplication.CreateBuilder()`, `builder.Build()`).
- Converting a function literal to an imported delegate type (`RequestDelegate`) and registering it through instance-style extension dispatch (`app.Run(handler)`).
- Calling an imported instance method with an explicit optional argument (`HttpResponse.WriteAsync(text, CancellationToken.None)`).

ASP.NET minimal-API endpoint routing (`app.MapGet("/", handler)`) is not used here: it relies on overload resolution converting a function literal directly to `System.Delegate` in argument position, which the GSharp compiler does not support yet. The terminal-middleware shape above is the idiomatic way to express a minimal ASP.NET Core app in GSharp today.
