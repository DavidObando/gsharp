# GSharp web app

Generated by `dotnet new gsharp-web`.

This minimal [ASP.NET Core](https://learn.microsoft.com/aspnet/core/) web application uses the modern `WebApplication` host (Kestrel) and minimal-API endpoint routing. It maps an HTTP GET endpoint for `/` that responds with `Hello from GSharp on ASP.NET Core!`. The handler is declared as a `Func[string]` value first so it binds cleanly to the delegate `MapGet` expects, then it is registered with `app.MapGet("/", hello)`. `app.Run(url)` starts Kestrel and blocks until the process is stopped.

The project references the shared `Microsoft.AspNetCore.App` framework via a `<FrameworkReference>` in `GsharpWebApp.gsproj`, so no extra NuGet packages are needed for the web stack.

## Build & run

```sh
dotnet build
dotnet run
```

Then open <http://localhost:5117/> in a browser or `curl` it. Press <kbd>Ctrl</kbd>+<kbd>C</kbd> to stop.

If port 5117 is already in use, change the URL in `Program.gs`.

## What the compiler supports today

This template demonstrates that GSharp can drive the ASP.NET Core minimal-API pipeline using language features that now work end to end:

- Constructing and configuring imported types (`WebApplication.CreateBuilder()`, `builder.Build()`).
- Binding a function literal to a delegate value (`Func[string]`) and registering it for a route through instance-style extension dispatch (`app.MapGet("/", hello)`).
- Starting the host with an imported instance method (`app.Run(url)`).

The handler is bound to a `Func[string]` value before being passed to `MapGet`, rather than written as a lambda literal in argument position: the GSharp compiler does not yet convert a function literal directly to a delegate while resolving an extension-method overload. Binding the handler to a typed value first is the idiomatic way to register minimal-API endpoints in GSharp today.
