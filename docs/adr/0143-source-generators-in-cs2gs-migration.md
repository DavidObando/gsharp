# ADR-0143: run Roslyn source generators during cs2gs migration

- **Status**: Proposed
- **Date**: 2026-07-06
- **Phase**: Phase 9 — migration completeness
- **Related**: ADR-0115 (cs2gs migration tool), ADR-0138 (construct coverage & gap triage), ADR-0142 (resx codebehind generator — wiring precedent), ADR-0027 (no Roslyn in the compiler), ADR-0144 (partial types — companion), ADR-0145 (native generator host — companion), issue [#2201](https://github.com/DavidObando/gsharp/issues/2201)

## Context

`cs2gs migrate` (ADR-0115) translates **hand-written** C# into G# but never runs
the compilation's Roslyn **source generators**. Any symbol a generator would
produce is therefore absent from the translated G#, and every reference to it
fails at the gsc compile stage (`GS0157`/`GS0159`). Discovered while migrating
the Oahu corpus, where two generator families dominate the failures:

1. **CommunityToolkit.Mvvm** — `[ObservableProperty]` on a field generates a
   public property with `INotifyPropertyChanged` wiring; `[RelayCommand]` on a
   method generates an `ICommand` property. The generated members are **partial
   parts of the user's own types** — they cannot live in a separate metadata
   assembly. This is the dominant root cause in `Oahu.UI` (45 compile errors).
2. **Nerdbank.GitVersioning `ThisAssembly`** — a class of version constants.
   This is **not** a Roslyn source generator: an MSBuild target writes a `.cs`
   file into `obj/` and adds it as a `Compile` item (design-time builds produce
   it too, which is how IDEs see `ThisAssembly`).

The current loader (`CSharpProjectLoader.BuildDocuments`) **skips** every
generated tree — anything under the project's own `obj/`/`bin/`, or carrying an
`<auto-generated>` header (`HasAutoGeneratedHeader`) — emitting an informational
`CS2GS0002`. That blanket skip is exactly why both families' output vanishes.

The binding constraint from ADR-0027 is unchanged and unthreatened here: Roslyn
stays out of the **compiler**. cs2gs already uses Roslyn as an *external,
offline C# front-end in a sibling process* (ADR-0115); running the project's
generators is more of that same use, in the same process, at the same stage.

A crucial enabler already exists: the translator merges **partial-type parts**
compilation-wide (issue #1910). `CollectPartialTypeParts`
(`CSharpToGSharpTranslator.cs`) walks every named type's
`DeclaringSyntaxReferences` and folds all parts into one merged G# type; it also
strips `global::`/alias qualifiers. So once generated trees are present in the
compilation and their documents are translated, MVVM's generated members fold
into the user's G# type almost for free — provided three hazards are handled
(primary-part ordering, partial-member shapes, and the resx carve-out).

## Decision

Run the compilation's source generators during the Translate stage and route
their output — plus MSBuild-generated `Compile` items like `ThisAssembly` —
through the existing translator, classified so that only type-bearing generated
documents are translated. Generators are run **generically** (whatever the
project references); no generator is special-cased.

### A. Explicit generator execution at load time (strip-then-run)

A new `SourceGeneratorRunner` (`Cs2Gs.Translator/Loading/`) runs between
`project.GetCompilationAsync()` and `BuildDocuments` inside
`CSharpProjectLoader.BuildLoadedProjectAsync`:

1. **Strip.** Remove any generator-produced trees the workspace may already
   have injected (`project.GetSourceGeneratedDocumentsAsync` →
   `compilation.RemoveSyntaxTrees(...)`). This makes step 3 the single
   authoritative generator pass, regardless of whether MSBuildWorkspace on
   Roslyn 5.6 ran generators implicitly (empirically the symbols are missing —
   see Consequences; the design is correct either way).
2. **Harvest generators.** For each `AnalyzerFileReference` in
   `project.AnalyzerReferences`, call `GetGenerators(LanguageNames.CSharp)`
   inside a per-reference `try`/`catch`. A load failure (bad image, Roslyn
   version mismatch, missing dependency) becomes a `CS2GS0004` **warning**
   naming the analyzer path — never an exception out of the loader. Sort the
   collected generators deterministically by `(assembly name, generator type
   full name)`.
3. **Run.** If any generators loaded,
   `CSharpGeneratorDriver.Create(generators, additionalTexts, parseOptions,
   optionsProvider).RunGeneratorsAndUpdateCompilation(stripped, out output,
   out _)`, forwarding `project.AnalyzerOptions.AdditionalFiles`, the project's
   `CSharpParseOptions`, and — importantly —
   `project.AnalyzerOptions.AnalyzerConfigOptionsProvider`, through which
   CommunityToolkit.Mvvm reads MSBuild-visible feature switches (e.g.
   `INotifyPropertyChanging` support).
4. **Isolate crashes.** The driver already traps generator exceptions.
   Inspect `driver.GetRunResult().Results`: any entry with `Exception != null`
   → a `CS2GS0004` diagnostic carrying the generator type + exception summary;
   that generator's output is simply absent and the pipeline continues (the
   downstream gsc `GS0157` then has the `CS2GS0004` artifact as its documented
   root cause). Generator-*reported* diagnostics of Warning severity or above
   fold into load diagnostics like ordinary compiler diagnostics.
5. **Record provenance.** From each `GeneratorRunResult.GeneratedSources`, map
   `SyntaxTree → GeneratedOrigin { GeneratorAssemblyName, GeneratorTypeName,
   HintName }` and register it in a `GeneratedSourceRegistry` (a
   `ConditionalWeakTable`-backed static) so translator-side code that sees only
   the `Compilation` can ask `IsGenerated(tree)`.

`SignificantDiagnostics` is then computed on the **output** compilation, so the
pre-existing "missing generated symbol" errors disappear once generation runs.
This is the actual fix for #2201.

### B. Classification: translate a generated document iff it declares a type

`BuildDocuments` replaces the blanket skip with a per-tree classification.
Origin is one of: `SourceGenerator` (registry hit), `BuildOutput` (path under
the project's own `obj/`/`bin/` — the `ThisAssembly` shape, handled
generically), `HeaderGenerated` (an `<auto-generated>` header outside
`obj`/`bin`), or hand-written.

- **Hand-written** → `LoadedDocument`, exactly as today.
- **Generated (any origin)**:
  - **Resx carve-out first.** If the file path equals the designer path of a
    discovered `.resx` (`ResxCodeGenerator.GetDesignerFilePath`), skip with
    `CS2GS0002` — ADR-0142's resx **generation** path owns that surface, and
    translating the hand-authored `Resources.Designer.cs` too would produce a
    duplicate-type collision. (This requires hoisting `DiscoverResxFiles` above
    `BuildDocuments` so the designer-path set is available; a small reorder.)
  - Else **translate** iff the root declares at least one named type at any
    depth (`BaseTypeDeclarationSyntax` or `DelegateDeclarationSyntax`), emitting
    a new `CS2GS0005` **info** diagnostic ("generated source included in
    translation: '{path}' ({origin})") as the decision trace.
  - Else **skip** with `CS2GS0002` (message extended: "declares no types"). This
    keeps `AssemblyInfo.cs`, `GlobalUsings.g.cs`, and `.AssemblyAttributes.cs`
    skipped, and un-skips `ThisAssembly.cs` (declares `internal class
    ThisAssembly`) with **zero** Nerdbank.GitVersioning-specific code.

Generic by construction: any MSBuild target injecting a typed `.cs` as a
`Compile` item, and any generator emitting types, flows through the same gate.

### C. Hand-written parts win primary

With generated trees now in the compilation, a generated part could sort first
and become the "primary" part (`parts[0]`) — the single declaration point that
actually emits the merged G# type — landing the whole type in a generated
document. `CollectPartialTypeParts` ordering gains a **non-generated-before-
generated** key, applied after the existing record-with-positional-parameters
key and before the file-path/span keys:

```
order by:
  1. record part carrying the positional parameter list   (unchanged)
  2. NON-generated before generated                        (new)
  3. file path (ordinal)                                   (unchanged)
  4. span start                                            (unchanged)
```

"Generated" here is `GeneratedSourceRegistry.IsGenerated(tree) ||
HasAutoGeneratedHeader(tree)` — the header sniff moves into a shared helper so
loader and translator agree, and the fallback covers in-memory tests that inject
pre-generated trees without a registry entry. Consequence: MVVM's generated
parts merge into the user's hand-written document via the existing #1910
machinery. If *every* part of a type is generated, `parts[0]` is generated and
the type correctly emits in that generated document.

A generated document whose every type is a **non-primary** partial part
translates to an empty compilation unit; `TranslateStage` **does not write** a
`.gs` for an empty generated-origin unit (no clutter, no naming-dedup pressure).
Hand-written documents keep today's behavior.

### D. Partial-member policy

The translator has **no** partial-member handling today: a declaration-only
`partial void OnNameChanged(...)` falls into the bodiless-method path and emits a
bodiless `func` inside a class (a shape gsc rejects), and a
defining+implementing pair emits **both** members (a duplicate). Both shapes
appear in real MVVM output. Rules, keyed on the Roslyn symbol
(`IMethodSymbol`/`IPropertySymbol` `.IsPartialDefinition` /
`.PartialImplementationPart` / `.PartialDefinitionPart`):

1. **Unimplemented partial method** (`IsPartialDefinition` with no
   implementation part) → **elide both the declaration and every call site**,
   each with an info diagnostic. This mirrors csc exactly: a deletable partial
   method is necessarily `void`, has no `out` parameters, and can only be called
   in statement position — so the compiled C# assembly contains neither the
   method nor its calls. Elision reproduces that; it also avoids flooding the
   frozen output with up to four dead hooks per `[ObservableProperty]`.
2. **Defining + implementing pair** (partial methods, and C# 13 partial
   properties — e.g. the user's `[ObservableProperty] public partial string
   Name { get; set; }` defining part paired with the generator's implementing
   part) → **translate only the implementing part**, via a shared
   `IsElidedPartialMember(member, semanticModel)` predicate applied at the
   merged-members choke point so the primary part and every merged-in part are
   filtered uniformly. When a dropped defining part carried attributes, report
   an info noting the attribute lives only in the frozen snapshot's C# metadata.

### E. Provenance and triage

- `LoadedDocument` gains a nullable `GeneratedOrigin` (generator assembly/type,
  hint name, or the obj-relative path for `BuildOutput`).
- `TranslateStage` prepends provenance to each generated-origin `.gs` via the
  existing `CompilationUnit.LeadingComments` (no printer change):

  ```
  // <auto-generated>
  // Translated by cs2gs from source-generator output:
  //   CommunityToolkit.Mvvm...ObservablePropertyGenerator, hint 'MainViewModel.g.cs'
  // Frozen at migration time (ADR-0143); regeneration in native G# builds is ADR-0145.
  ```
- `EmittedGsFile` and the `TranslationUnsupported` triage artifact carry the
  origin; a new `TriageBuilder.GeneratorFailure(...)` records `CS2GS0004`,
  fingerprinted on **generator identity** (not exception text) for rerun
  stability. Fingerprints for unsupported-construct artifacts stay
  shape-normalized, so the same gap in generated vs hand-written code dedupes to
  one ledger entry — but the origin is recorded in the notes, because a
  generated-origin gap is **by definition** a cs2gs/gsc gap (the user cannot
  edit the source), never "the user should rewrite it".

### F. Determinism

Generators run in a deterministically sorted order against fixed parse options;
`GeneratorDriver` output ordering is itself deterministic — so identical inputs
produce byte-identical emitted `.gs` across pipeline reruns (ADR-0115's
requirement). `ThisAssembly` content is deterministic **per git commit** but
changes across commits, so the checked-in corpus must **not** take a real
Nerdbank.GitVersioning dependency (its baseline would churn every commit); the
generic obj-`Compile`-item path is exercised by a fixture with **fixed** content
instead (§G). For real migrations, version constants freeze at the snapshot
commit — see Scope limitation.

### G. Testing surface

- `CSharpProjectLoader.LoadInMemory` gains overloads accepting
  `IIncrementalGenerator` instances (wrapped via `.AsSourceGenerator()` and run
  through the shared `SourceGeneratorRunner` core) and pre-generated
  `(fileName, source, origin)` tuples, so the generator flow is unit-testable
  without MSBuild.
- A new corpus app `tools/cs2gs/corpus/L6-Mvvm-Console/` restores a pinned
  `CommunityToolkit.Mvvm`, declares a viewmodel with `[ObservableProperty]`
  fields (including a **user-implemented** `OnNameChanged` hook, exercising rule
  D.2) and a `[RelayCommand]` method, prints a deterministic `PropertyChanged`
  sequence to a golden baseline, and carries a `BeforeCompile` target writing a
  **fixed-content** `obj/BuildInfo.g.cs` (the generic `ThisAssembly`-shaped
  path). The full pipeline (Translate → Compile → IlVerify → TestParity) must
  pass, or its fingerprints be ledgered in the same change per ADR-0138; reruns
  must be byte-identical.

## Consequences

- Positive: closes the dominant `Oahu.UI` blocker — MVVM `[ObservableProperty]`
  / `[RelayCommand]` members and NB.GV `ThisAssembly` now resolve in translated
  G# — without special-casing any generator.
- Positive: gsc gaps *inside* real generated code now surface as ledgered,
  generated-origin triage artifacts (the intended ADR-0138 loop), correctly
  attributed as tool gaps rather than user-rewritable.
- Neutral: generated members are **frozen** as ordinary translated G# source;
  see Scope limitation.
- Negative: generator execution runs per project (and per transitive sibling in
  `LoadProjectWithReferencesAsync`), adding load time proportional to the
  generators referenced.
- Constraint on ADR-0144: once G# has partial types, cs2gs could emit one G#
  `partial` part per C# part and retire most of the #1910 merge machinery
  (including Decision C's ordering) — a future simplification, explicitly **not**
  blocking this ADR.

## Scope limitation (explicitly out of scope)

Generated members are frozen at migration time as ordinary G# source; nothing
regenerates them in a native G# build — that is ADR-0145 (native generator
host). Nerdbank.GitVersioning version constants therefore freeze at the snapshot
commit. The originating package references remain required at gsc compile/run
time for the generators' **runtime** types (`ObservableObject`,
`IRelayCommand`), which already flow through `TranslateStage`'s external
reference capture; the compile-time-only attributes translate inertly on the
frozen members. The project must have been restored (and, for obj-`Compile`-item
inputs, design-time-built) before `cs2gs migrate` — already true of
MSBuildWorkspace loading generally.

## Alternatives considered

- **Rely on MSBuildWorkspace's implicit generator run and patch the gaps.**
  Rejected: empirically the symbols are missing (#2201), and depending on
  undocumented workspace behavior across Roslyn versions is fragile. Strip-then-
  run is authoritative regardless of what the workspace did.
- **Special-case the well-known generators** (synthesize MVVM members and
  `ThisAssembly` from templates). Rejected: brittle, unbounded (every new
  generator family is new code), and duplicates logic the real generators
  already implement correctly. The type-declares gate un-skips `ThisAssembly`
  generically with no NB.GV-specific code.
- **Synthesize empty bodies for unimplemented partial-method hooks** instead of
  eliding. Rejected: it diverges from what csc emits and floods the frozen
  output with up to four dead members per `[ObservableProperty]`. Elision is
  both simpler at the source level and faithful to the compiled assembly.
- **Translate every generated file, including attribute-only ones.** Rejected:
  `AssemblyInfo.cs` / `GlobalUsings.g.cs` carry no types and translating them
  adds noise and duplicate-attribute hazards; the type-declares gate is the
  precise line.
- **Block on ADR-0144 partial types and emit parts directly.** Rejected as the
  *first* step: the #1910 merge already works and unblocks Oahu now; emitting
  parts becomes a clean follow-up once 0144 lands.
