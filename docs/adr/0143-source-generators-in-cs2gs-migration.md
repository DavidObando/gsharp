# ADR-0143: generated code in cs2gs migration ‚Äî reproduce at build, don't freeze

- **Status**: Proposed
- **Date**: 2026-07-06
- **Phase**: Phase 9 ‚Äî migration completeness
- **Related**: ADR-0115 (cs2gs migration tool), ADR-0145 (native generator host ‚Äî reproduces Roslyn generators at build), ADR-0144 (partial types), ADR-0142 (resx codebehind generator ‚Äî the "reproduce natively in the G# build" precedent), ADR-0138 (construct coverage & gap triage), ADR-0027 (no Roslyn in the compiler), issues [#2201](https://github.com/DavidObando/gsharp/issues/2201), [#2209](https://github.com/DavidObando/gsharp/issues/2209), [#2210](https://github.com/DavidObando/gsharp/issues/2210), [#2211](https://github.com/DavidObando/gsharp/issues/2211)

## Context

`cs2gs migrate` (ADR-0115) translates **hand-written** C# to G# and **skips**
every generated tree ‚Äî anything under the project's own `obj/`/`bin/`, or
carrying an `<auto-generated>` header (`CSharpProjectLoader.BuildDocuments` ‚Üí
`HasAutoGeneratedHeader`, emitting `CS2GS0002`). Discovered while migrating the
Oahu corpus: because the generated code is skipped, every symbol it would
produce is missing and references to it fail at the gsc compile stage
(`GS0157`/`GS0159`). Two families dominate `Oahu.UI`:

1. **CommunityToolkit.Mvvm** ‚Äî `[ObservableProperty]`/`[RelayCommand]` generate
   members *into the user's own `partial` types* (the dominant blocker).
2. **Nerdbank.GitVersioning `ThisAssembly`** ‚Äî a class of version constants; not
   a Roslyn generator but an MSBuild target that writes a `.cs` into `obj/`.

An earlier draft of this ADR proposed the obvious fix: run the compilation's
generators *inside cs2gs* and translate/freeze their output into the migrated
G#. That predates two things that have since shipped, and it turns out to be the
wrong shape:

- **ADR-0145 (gsgen) now runs Roslyn generators in the G# build itself.** A
  native `.gsproj` that references a generator package resolves and runs the
  generator (via gsgen) before gsc, exactly as C# does. A *migrated* project that
  keeps the package reference, the annotated members' attributes, and the
  `partial` modifier therefore **regenerates those members on every build** ‚Äî no
  frozen copy required.
- **The G# SDK/gsc already reproduce build-derived facts from project
  properties.** Assembly version/identity attributes are stamped by gsc from the
  values the SDK passes (the resx codebehind generator, ADR-0142, is the
  precedent for reproducing a *file's* generated content natively in the G#
  build rather than translating it).

So the migration does not need to run generators or freeze their output. It
needs to **preserve the inputs that drive regeneration** and **stop dropping the
hand-authored content it currently skips**. This ADR reframes the problem from
"translate generated code" to "reproduce generated code in the migrated
project's own build, and translate only what is genuinely hand-authored."

The ADR-0027 boundary is unaffected: no Roslyn enters the compiler. Generators
run in gsgen (a sibling tool); gsc gains only the small, self-contained
`ThisAssembly` synthesis below.

## Decision

### A. Principle: reproduce at build, don't freeze

cs2gs does **not** run source generators and does **not** translate generated
output into the migrated G#. Every category of generated code is reproduced by
the migrated project's own G# build; cs2gs's job is to (1) translate the
hand-authored source, (2) preserve the project-level inputs that drive
regeneration (package references, attributes, `partial`, version/identity
properties, implicit-using items), and (3) drop the generated artifacts so the
build's fresh output is the single source of truth. The result is faithful,
never stale, and a fraction of the code the freeze approach required.

### B. Roslyn `[Generator]` output ‚Üí reproduced by gsgen (ADR-0145)

For the MVVM family and every other Roslyn generator, migration relies on the
gsgen build path. cs2gs must only keep the trigger intact, which it already
largely does:

- **Attributes survive.** `[ObservableProperty]`/`[RelayCommand]` translate to
  `@ObservableProperty`/`@RelayCommand` on the migrated member.
- **The augmented type is `partial`.** The translator's `preservePartialParts`
  mode emits a C# `partial` type as a G# `partial` part, so gsgen's generated
  part can augment it (all parts of a partial type must be marked `partial`, or
  gsc raises `GS0475`).
- **The package reference carries into the `.gsproj`**, and the SDK's
  `_GsharpResolveAnalyzers` (ADR-0145 ¬ßE) resolves the generator's analyzer
  assets. At build, gsgen's stub projection + `ForAttributeWithMetadataName`
  reproduce the members. (This is exactly the loop ADR-0145's end-to-end test
  proves for `@ObservableProperty`.)

**File/options-driven generators (Avalonia `.axaml`, issue #2223).** Some
generators consume non-source inputs rather than attributes ‚Äî Avalonia's XAML
name generator reads each `.axaml` (a Roslyn `AdditionalText`) and, guided by
`build_property.AvaloniaNameGeneratorBehavior` +
`build_metadata.AdditionalFiles.SourceItemGroup=AvaloniaXaml`, generates the
partial `InitializeComponent()` and named-control fields into the `.axaml.cs`
code-behind. These reproduce through the same gsgen path once gsgen forwards
`AdditionalText` + `AnalyzerConfigOptions` (ADR-0145 ¬ßC, issue #2223). cs2gs
keeps the trigger intact by (a) discovering the project's `@(AdditionalFiles)` +
`.axaml` (`CSharpProjectLoader`), (b) forwarding them ‚Äî tagged
`SourceItemGroup=AvaloniaXaml` ‚Äî plus `RootNamespace`/`ProjectDir` options to the
Compile stage's gsc invocation, and (c) marking the code-behind `partial` (the
project's analyzer references already trigger `markMergedTypePartial`). The
runtime side (Avalonia's `CompileAvaloniaXaml` IL-rewrite) is a language-agnostic
post-compile MSBuild task that hooks the standard `AfterCompile` chain and needs
no G# changes.

**Pipeline requirement.** The cs2gs Compile stage shells out to gsc directly
(`GscInvoker`, not the SDK), so gsgen does not run there today and the stage
would still see missing symbols. The Compile stage must run gsgen before gsc ‚Äî
either by building the emitted `.gsproj` through `Gsharp.NET.Sdk` (which runs
gsgen) or by invoking the `gsgen` CLI to produce the `.g.gs` and passing them
alongside the user `.gs` ‚Äî so validation matches the project's real build.

### C. `ThisAssembly` and version constants ‚Üí synthesized by the G# build

Instead of translating Nerdbank.GitVersioning's `obj/ThisAssembly.cs`, the G#
build **synthesizes** the `ThisAssembly` type from the version/identity facts it
already receives (gsc is passed the assembly version by the SDK's build task;
this extends that to the small set of `ThisAssembly` constants ‚Äî
`AssemblyVersion`, `AssemblyFileVersion`, `AssemblyInformationalVersion`, and
peers ‚Äî sourced from the build's version inputs, e.g. `version.json`). This
mirrors ADR-0142's resx codebehind generator: a build-derived file is
reproduced natively in the G# build rather than frozen at migration. cs2gs drops
the generated `ThisAssembly.cs`; references to `ThisAssembly.*` resolve at build
time and stay correct across commits (a frozen copy would go stale every
commit).

### D. SDK-derived non-type files ‚Üí reproduced from project properties

`AssemblyInfo.cs` (assembly identity/version attributes) and `GlobalUsings.g.cs`
(implicit global usings) are generated by the .NET SDK from project properties ‚Äî
`<Version>`/`<AssemblyTitle>`/‚Ä¶ and `ImplicitUsings`/`<Using>` items. cs2gs
preserves those properties/items in the emitted `.gsproj` and drops the
generated files; the G# SDK/gsc reproduce the equivalent assembly attributes and
global imports. Nothing is translated or frozen for these.

### E. Classify by provenance, not by type-content ‚Äî and translate the
hand-authored non-type source cs2gs currently drops

The blanket skip is replaced by a single question per generated candidate:
**does the migrated project's build reproduce this file?**

- **Reproduced by the build** ‚Üí *drop it* (with an informational `CS2GS0002`):
  Roslyn-generator output (¬ßB), synthesized `ThisAssembly` (¬ßC), SDK-derived
  `AssemblyInfo`/`GlobalUsings` (¬ßD). No frozen copy.
- **Hand-authored** ‚Üí *translate it*, **including files that declare no types**.
  The current "generated" heuristic over-collects: a hand-written source that
  carries only *assembly-level* content ‚Äî `[assembly: InternalsVisibleTo(...)]`,
  custom assembly attributes, hand-written `global using` directives ‚Äî declares
  no type and is silently dropped today, losing real semantics. cs2gs must
  translate this content: assembly-level attributes ‚Üí G# assembly-level
  annotations (G# already has the surface ‚Äî `CompilationUnitSyntax.AssemblyAttributes`
  / `ParseFileLevelAssemblyAttributes`), hand-written global usings ‚Üí G#
  `import`s. So this is a translator/classification change, not a new
  language feature. The axis that matters is **provenance (reproducible-by-build vs.
  hand-authored)**, not whether the file happens to declare a type ‚Äî the old
  type-declares gate conflated a generated `GlobalUsings.g.cs` (drop) with a
  hand-written `AssemblyAttributes.cs` (translate).

The resx carve-out is unchanged: a `Resources.Designer.*` is owned by ADR-0142's
resx generation path and is dropped here so the two don't collide.

### F. Provenance, triage, and determinism

- **Provenance for the residue.** Any generated file that is *neither*
  reproduced by the build *nor* hand-authored ‚Äî an exotic MSBuild-emitted
  `Compile` item, a T4 template output with no G# equivalent ‚Äî is reported (a
  `CS2GS0004`-class triage artifact, ADR-0138) so it surfaces rather than
  vanishing. These are, by definition, tool/SDK gaps (the user cannot edit the
  source), and the triage note says so.
- **gsgen fidelity is where the real gaps now live.** Because migration relies
  on gsgen to reproduce Roslyn output, the quality of that back-translation is
  the gating factor. The known gaps against real MVVM output are already filed:
  gsc static access through a fully-qualified generic type
  ([#2209](https://github.com/DavidObando/gsharp/issues/2209)), gsc unqualified
  calls to inherited imported-base members
  ([#2210](https://github.com/DavidObando/gsharp/issues/2210)), and cs2gs/gsgen
  import synthesis for generator output
  ([#2211](https://github.com/DavidObando/gsharp/issues/2211)). Fixing these
  benefits native builds and migration identically.
- **Determinism** is inherent: reproduction runs the real generator / the SDK's
  own derivation on every build, so the migrated project is byte-stable per its
  inputs without cs2gs freezing anything.

## Consequences

- Positive: closes the dominant `Oahu.UI` blocker while *removing* the largest
  pieces the freeze approach required ‚Äî no in-cs2gs generator driver, no
  generated-tree translation, no primary-part ordering hazard, no frozen-output
  determinism fixtures. cs2gs shrinks; the migrated project's build carries the
  weight.
- Positive: generated members stay **live and correct** ‚Äî they regenerate when
  the package, the annotations, or the version source change, matching how the
  project behaves as native G#. No stale frozen snapshots.
- Positive: one code path serves native builds and migration. The gsgen fidelity
  fixes (#2209‚Äì#2211) and a G#-native `ThisAssembly` are reusable build-time
  capabilities, not migration-only translator special-cases.
- Neutral ‚Üí new work moves to the build system: the cs2gs pipeline's Compile
  stage must run gsgen (¬ßB); gsc/SDK gains `ThisAssembly` synthesis (¬ßC); G#
  needs an assembly-attribute surface for ¬ßE if it lacks one.
- Negative: a migrated project is **not fully self-contained** ‚Äî it keeps
  depending on the generator package's analyzer (and gsgen) at build time. For
  MVVM this is moot (the runtime `ObservableObject` base already ties the project
  to the package), but a "port away from all C#/NuGet" migration would want a
  freeze mode (see Scope limitation).

## Scope limitation (explicitly out of scope)

- **A self-contained "freeze" migration mode** ‚Äî translating Roslyn-generator
  output into frozen, hand-editable G# so the project can shed the generator
  package entirely ‚Äî is a possible opt-in but is *not* the default and not built
  here. The default is live reproduction (¬ßA).
- **Non-reproducible generated code** ‚Äî MSBuild codegen with no G# analogue, or
  generators outside gsgen's attribute-driven fidelity boundary (ADR-0145 ¬ßH) ‚Äî
  is surfaced via triage (¬ßF), not silently frozen. Closing each is its own
  gsc/SDK/gsgen item.

## Alternatives considered

- **Run source generators inside cs2gs and translate/freeze their output** (this
  ADR's original draft). Rejected: it duplicates the generator execution and
  back-translation that ADR-0145 already performs in the build, freezes
  stale snapshots that drift from the package/version source, and carries a large
  translator surface (a strip-then-run driver, a primary-part ordering fix, a
  generated-tree classification, frozen-output determinism fixtures) ‚Äî nearly all
  of which the reproduce-at-build principle deletes.
- **Special-case the well-known generators** (synthesize MVVM members /
  `ThisAssembly` from templates in cs2gs). Rejected: brittle and unbounded, and
  it re-implements logic the real generator / SDK already own. ¬ßB/¬ßC reproduce
  them generically.
- **Keep the type-declares skip gate.** Rejected: it classifies on the wrong
  axis. It correctly drops a generated `GlobalUsings.g.cs` but also silently
  drops a hand-written `AssemblyAttributes.cs`, losing real semantics. Provenance
  (¬ßE) is the correct gate.
- **Leave `ThisAssembly` frozen (translate the obj `.cs`).** Rejected: the
  constants go stale every commit, and a build-native synthesis (¬ßC) ‚Äî following
  the ADR-0142 resx precedent ‚Äî keeps them correct with no migration-time work.
