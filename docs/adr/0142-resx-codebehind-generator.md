# ADR-0142: resx codebehind generator (`Resources.Designer.gs`)

- **Status**: Accepted
- **Date**: 2026-08-14
- **Phase**: Phase 9 — language surface completeness
- **Related**: ADR-0115 (cs2gs migration tool), issue [#2200](https://github.com/DavidObando/gsharp/issues/2200)

## Context

A `.resx` resource file's strongly-typed accessor class
(`Resources.Designer.cs`) is produced by Visual Studio's `ResXFileCodeGenerator`
custom tool whenever the `.resx` is saved; it is never hand-written and is
therefore excluded from `cs2gs` translation as generated source (its
`<auto-generated>` header trips `CSharpProjectLoader.HasAutoGeneratedHeader`).
Skipping it left a hole: migrated code that references the generated class
(directly, or through a `using R = ...Resources;` alias) fails to resolve
(`GS0157`), cascading into `GS0159` errors for every member access through that
alias. The G# language server has the same gap for any `.gsproj` that embeds a
`.resx` directly (rather than via C# migration) — nothing regenerates its
codebehind when the `.resx` is created or edited.

## Decision

### A. One shared generator, `GSharp.Core.Resx`

A single library (`src/Core/Resx/`) parses a `.resx` and renders its G#
designer class, mirroring the shape of `ResXFileCodeGenerator`'s C# output
(a `ResourceManager`/`Culture` pair plus one property per resource) but in G#
syntax:

- `ResxDocument` parses the resx XML (`System.Xml.Linq`), skipping
  `>>`-prefixed designer metadata rows and the `resheader` block.
- `ResxIdentifier` sanitizes a resource key into a valid G# identifier,
  resolving reserved-keyword collisions (`class` → `class1`) with a numeric
  suffix — G# has no verbatim-identifier escape (no `@keyword`).
- `ResxTypeNameMapper` maps an assembly-qualified CLR type name (as resx
  stores it) to a G# type clause, including primitive keyword spellings
  (`System.Byte` → `uint8`) and G#'s prefix array spelling (`[]uint8`, not
  `uint8[]`).
- `ResxDesignerGenerator` renders the full source: `internal init() { }` (the
  constructor is always `internal`, matching the real generator's
  instantiation-discouraged, all-members-static design, regardless of the
  class's own accessibility), a `shared { }` block with `ResourceManager` and
  `Culture` properties, and one property per entry — an expression-bodied
  `GetString` call for string resources, a `(... as T)!!` cast over
  `GetObject` for everything else.
- `ResxCodeGenerator` is the public facade (`GenerateFromFile`,
  `GetDesignerFilePath`) both consumers below call through, so they can never
  drift from each other.

The library lives in `Core` rather than a new project because `Core` is
already a transitive dependency of both the language server and
`Cs2Gs.Translator` (via `Cs2Gs.CodeModel`); no new project/solution wiring is
needed. All CLR type references in generated code are fully-qualified dotted
names (`System.Resources.ResourceManager`), so the generator never has to
reason about import ambiguity.

### B. Namespace convention

The generated class's namespace is the project's root namespace plus one
dotted segment per folder between the project root and the `.resx` file
(`Properties/Resources.resx` under root namespace `App` → `App.Properties`),
matching MSBuild's default resx-namespace convention (no `CustomToolNamespace`
override).

### C. Language server: regenerate on save

The language server resolves each discovered project's `RootNamespace`
(`ProjectDiscovery.ResolveRootNamespace`, falling back to `AssemblyName`) and
watches `.resx` files the same way it already watches `.gs`/`.gsproj`
(`serverManager.ts`'s file-system-watcher glob gained `resx`). On a
`Created`/`Changed` `workspace/didChangeWatchedFiles` notification for a
`.resx` inside a project's scope, `LspServer.HandleResxFileChange` calls
`ResxCodeGenerator.GenerateFromFile` and writes/registers the resulting
`{Base}.Designer.gs` next to the `.resx`. A `Deleted` notification is a no-op
— matching Visual Studio, which never deletes a previously generated
`Designer.cs` when its `.resx` is removed. Any generation failure (malformed
XML, IO error) is caught and logged; it never crashes the LSP.

### D. cs2gs: generate instead of translate

`CSharpProjectLoader` discovers every `.resx` under a loaded project's
directory (excluding `obj`/`bin`) via filesystem glob — Roslyn's `Project`
does not surface `.resx` as a document; it is an MSBuild `EmbeddedResource`
item, not `AdditionalFiles`. It also resolves the project's effective root
namespace via Roslyn's `Project.DefaultNamespace` (falling back to
`AssemblyName`, then the project file's base name). `TranslateStage` then
calls `ResxCodeGenerator.GenerateFromFile` for each discovered `.resx` and
writes the resulting designer `.gs` alongside the run's other translated
output, registering it as an `EmittedGsFile` so it flows into the Compile
stage exactly like a translated document. The pre-existing skip of a
hand-authored `Resources.Designer.cs` (as generated source) is unchanged —
this generation path is what now fills the gap that skip used to leave open.
A malformed `.resx` is caught and reported as a triage artifact
(`CS2GS0003`) rather than crashing the stage.

## Scope limitation (explicitly out of scope)

Embedding the compiled `.resources` manifest stream into the output assembly
(the piece that makes `ResourceManager.GetString`/`GetObject` succeed at
*runtime*) is **not** implemented here. This ADR only fixes the compile-time
blocker (`GS0157`/`GS0159`): generated code compiles, but running it throws
`System.Resources.MissingManifestResourceException` until a follow-up adds
SDK-level resource embedding to `gsc`/the `.gsproj` build.
